# Cleanup Classification Reference

Use this reference when deciding whether a path is safe to remove, must be backed up, or needs user confirmation.

## Usually Safe After Dry-Run Review

These are generally rebuildable when they are inside the project workspace and ignored by Git:

- Dependency installs: `node_modules`, `.pnpm-store`, `vendor/bundle`, `.venv`, `venv`, `.tox`
- Build outputs: `target`, `dist`, `build`, `out`, `.next`, `.nuxt`, `.svelte-kit`, `coverage`
- Tool caches: `.turbo`, `.vite`, `.parcel-cache`, `.pytest_cache`, `.mypy_cache`, `__pycache__`
- Runtime/test output: `output`, `diagnostics`, `logs`, `tmp`, `temp`, browser profiles from Playwright/Edge/Chrome tests
- Package temp logs: `.packaging-logs`, `.codex-logs`, `*.log` generated by local dev commands
- Empty probe directories after confirming they contain zero files and no nested directories with user data

Prefer `git clean -ndX` then `git clean -fdX` for this class.

## Back Up Or Archive First

These may contain useful work even when they look messy:

- Untracked source files, scripts, docs, tests, config, migrations, or assets
- Dirty tracked files and deleted tracked files
- Registered Git worktrees and nested repositories
- Old product folders when the project changed names or moved roots
- Release packages, installers, sidecar binaries, generated assets, or screenshots not confirmed uploaded
- Local database dumps, recordings, transcripts, corpora, or benchmark fixtures
- Local validation or truth-data folders such as `.runtime`, `test_dataset`, `manual_frontend_runs`, `manual_program_runs`, `raw_documents`, mailbox inventories, invoice extraction outputs, and audit ledgers
- Project-local agent context such as `.codex`, `.codegraph`, `.gstack`, `.claude`, or worktree metadata unless explicitly confirmed to be disposable

Create a backup outside the workspace before removing or restructuring these.

## Keep By Default

Do not classify these as clutter without strong project-specific evidence:

- `.git`, `.github`, `.gitignore`, `.gitattributes`, license files
- Source directories and app entrypoints
- Manifests and lockfiles: `package.json`, `package-lock.json`, `pnpm-lock.yaml`, `Cargo.toml`, `Cargo.lock`, `pyproject.toml`, `requirements.txt`, `go.mod`
- Build and release scripts, CI workflows, tests, fixtures, migrations, public assets
- Documentation that is part of the current project or shipped package

## Explicit Confirmation Required

Ask or clearly restate the user's explicit permission before deleting:

- Local ML/ASR/LLM models and datasets
- Installers, signed packages, release artifacts, or app bundles
- `.env`, credentials, certificates, keys, secrets, browser profiles, user data
- Release archives and app bundles under `dist`, `release-*`, `*.zip`, `*.exe`, `*.msi`, `*.dmg`, `*.pkg`, or signed package directories unless there is evidence they are rebuildable and already published elsewhere
- Anything outside the workspace
- Files locked by processes unless the process command line clearly points to a disposable dev server inside the workspace

## Evidence Rules

- A folder name matching an old product is evidence, not proof.
- A large size is evidence, not proof.
- A Git-ignored path is usually rebuildable, but still review the dry-run.
- A Git-ignored path containing local data, validation evidence, release artifacts, or agent context is not safe by default.
- A release package can move from "needs confirmation" to "delete after confirmation" when the local file name, size, and SHA256 match a published release asset or another trusted archive, and the build scripts/manifests show how to regenerate the unpacked staging tree.
- A tracked `build/` directory often contains packaging source such as specs, manifests, icons, and installer scripts. Only ignored subpaths such as `build/work`, `build/runtime`, and generated metadata are usually safe.
- A clean worktree whose branch is merged into the canonical branch can usually be removed with `git worktree remove`.
- An unmerged branch or dirty worktree is not safe to delete without backup and confirmation.
